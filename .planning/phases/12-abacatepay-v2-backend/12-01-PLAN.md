1. `BASE_URL` = `https://api.abacatepay.com/v2`

2. `abacateRequest<T>(path, body)` — same pattern but support both POST (body) and GET (no body, append query params). For GET requests used by product lookup, add a second overload or use method parameter.

4. `resolveProductId(packType: PackType): Promise<string>` — Lazy-fetch with in-memory cache. On first call for a packType, GET `/products/get?externalId=mf_{packType}`, extract `data.id` (the `prod_xxx`), cache in module-level `Map<string, string>`. On subsequent calls, return from cache. This avoids hardcoding prod_xxx IDs that differ between dev/prod (per research Pitfall 3).

5. `createCheckout(params: { productId: string; customerId: string; externalId: string; returnUrl: string; completionUrl: string }): Promise<{ id: string; url: string }>` — POST `/checkouts/create` with:

**Critical change — operation order (per research Pitfall 4):**

Current: createBilling -> create Payment -> return URL
New: create Payment (pending) -> resolveProductId -> createCheckout(externalId=payment.id) -> update Payment with checkoutId -> return URL

**Specific changes:**

1. Update imports: replace `createBilling` with `createCheckout, resolveProductId` from abacatepay. Add `ABACATEPAY_EXTERNAL_IDS` from credit-packs.

2. Customer creation: remove CPF gate. Only email is required for AbacatePay v2 customer (per research Pitfall 5). Still save CPF in our DB if provided, still create AbacatePay customer if profile lacks `abacatepayCustomerId`, but only pass email (and optionally name). Do NOT block on missing CPF.

```typescript
if (!customerId) {
  const abacateCustomerId = await createCustomer(user.email, user.name);
  customerId = abacateCustomerId;
  await prisma.userProfile.update({
    where: { clerkUserId: user.id },
    data: {
      abacatepayCustomerId: customerId,
      ...(data.cpf && { cpf: data.cpf }),
    },
  });
}
```

3. Create Payment FIRST (pending), then checkout:

```typescript
const payment = await prisma.payment.create({
  data: {
    clerkUserId: user.id,
    readingId: data.reading_id,
    packType: data.pack_type,
    amountCents: pack.price_cents,
    status: "pending",
  },
});

const productId = await resolveProductId(data.pack_type as PackType);
const baseUrl = process.env.NEXT_PUBLIC_BASE_URL;

const checkout = await createCheckout({
  productId,
  customerId,
  externalId: payment.id,
  returnUrl: `${baseUrl}/creditos`,
  completionUrl: data.reading_id
    ? `${baseUrl}/ler/resultado/${data.reading_id}?paid=1`
    : `${baseUrl}/conta/leituras?purchased=1`,
});

await prisma.payment.update({
  where: { id: payment.id },
  data: { abacatepayCheckoutId: checkout.id },
});
```

4. Return `{ checkout_url: checkout.url }`.

5. Update logger: replace `billingId` with `checkoutId` in log context.

6. Zod schema stays the same (pack_type, cpf optional, reading_id optional).

Keep rate limiting, error handling, and response format identical.
