**Rewrite `src/app/api/webhook/abacatepay/route.ts`:**

1. **Signature validation (per PAY-04, research Pitfall 2):**
   - Read raw body via `req.text()`
   - Get signature from header `x-webhook-signature` (Next.js lowercases headers, so use `.get("x-webhook-signature")`)
   - Call `verifyWebhookSignature(rawBody, signature)` from abacatepay.ts
   - Return 401 if invalid

2. **Event filtering:**
   - Parse body as JSON
   - If `body.event !== "checkout.completed"`, return 200 `{ ok: true }` (ignore non-checkout events)
   - Log `devMode` for observability

3. **Payment lookup (per PAY-04, research Pitfall 4):**
   - Extract `externalId` from `body.data.externalId` — this is our `payment.id`
   - Find payment: `prisma.payment.findUnique({ where: { id: externalId } })`
   - Return 404 if not found

4. **Idempotency check (per PAY-05):**
   - If `payment.status === "paid"`, log and return 200 (skip)

5. **Payment method:**
   - Extract from `body.data.payerInformation?.method` (per research), default to `"pix"` if absent

6. **Atomic transaction (per PAY-05) — PRESERVE existing logic:**
   The transaction body stays almost identical to current code. Changes:
   - `method` from `body.data.payerInformation?.method` instead of `body.data.method`
   - Everything else (mark paid, create credit_pack, debit if reading, upgrade tier, mark lead converted) stays the same

7. **Post-transaction:**
   - Logger with `paymentId`, `checkoutId` (from `body.data.id`), `packType`
   - Send email non-blocking (same as current)

**Type the webhook payload loosely** — per docs/abacatepay-v2.md best practices: "NAO usar Zod pra validacao estrita do payload — AbacatePay pode adicionar campos novos." Use a TypeScript interface for type hints but parse with `JSON.parse` and access fields with optional chaining.
