  <action>
Both pages are wrapped in Suspense already (LoginInner / RegistroInner), so useSearchParams() is safe to call inside the inner component.

In BOTH pages (login and registro):

1. Add `useSearchParams` import from `"next/navigation"` (alongside existing `useRouter`).
2. Inside the inner component, read: `const searchParams = useSearchParams(); const returnTo = searchParams.get("return") ?? null;`
3. Import `consumeCheckoutIntent` from `"@/lib/checkout-intent"`.
4. Create a helper `getRedirectUrl(): string` that:
   - Calls `consumeCheckoutIntent()` — if it returns an intent with `pacoteId`, return `/creditos?pacote=${intent.pacoteId}`
   - Else if `returnTo` is a non-null string starting with `/` (never redirect to external URLs), return `returnTo`
   - Else return `"/conta/leituras"`
5. Replace every `router.push("/conta/leituras")` with `router.push(getRedirectUrl())`.

For Google OAuth in BOTH pages (handleGoogle):

- Pass `returnTo` through the SSO callback so it survives the redirect. Read intent ONCE before building the URL (consumeCheckoutIntent removes from sessionStorage on call):
  ```typescript
  const intent = consumeCheckoutIntent();
  const redirectUrlComplete = intent
    ? `/creditos?pacote=${intent.pacoteId}`
    : returnTo && returnTo.startsWith("/")
      ? returnTo
      : "/conta/leituras";
  await signIn.authenticateWithRedirect({
    strategy: "oauth_google",
    redirectUrl: "/sso-callback",
    redirectUrlComplete,
  });
  ```
  (Same pattern in registro with `signUp.authenticateWithRedirect`.)

Keep `@clerk/nextjs/legacy` import — do NOT migrate to `@clerk/nextjs` here. That is Phase 11 (CLEAN-04).

Keep `<div id="clerk-captcha" />` exactly where it is in both forms — it is already present and correct (per FLOW-04 scope note).

<task type="auto">
  <name>Task 2: Add gender toggle to logged-in "pra mim" flow in /ler/nome</name>
  <files>src/app/ler/nome/page.tsx</files>
  <action>
Currently in the logged-in form (`isLoggedIn` branch), the Ela/Ele toggle only renders inside `{!isSelf && (...)}`. When `isSelf = true`, gender defaults to "female" with no way to change it.

Fix: Move the gender toggle OUTSIDE of the `{!isSelf && (...)}` block so it always renders in the logged-in form, regardless of the isSelf value.

Specific change — in the logged-in form section (around line 550-583), the current structure is:

```
{!isSelf && (
  <>
    <Input label="Nome da pessoa" ... />
    <div> Ela / Ele toggle </div>
  </>
)}
```

Change to:

```
{!isSelf && (
  <Input label="Nome da pessoa" ... />
)}

{/* Gender toggle — always visible (pra mim or outra pessoa) */}
<div className="flex flex-col gap-2">
  <span className="font-raleway text-[13px] text-bone tracking-[0.02em]">
    Essa leitura é para
  </span>
  <div className="flex gap-3">
    <ToggleButton
      selected={gender === "female"}
      onClick={() => setGender("female")}
      ariaLabel="Leitura para ela (feminino)"
    >
      Ela
    </ToggleButton>
    <ToggleButton
      selected={gender === "male"}
      onClick={() => setGender("male")}
      ariaLabel="Leitura para ele (masculino)"
    >
      Ele
    </ToggleButton>
  </div>
</div>
```

The label "Essa leitura é para" works for both "pra mim" and "pra outra pessoa" contexts. For "pra mim" the user picks their own gender; for "pra outra pessoa" they pick the other person's gender — same semantics as before.

Do NOT change any other logic. The `gender` state, `ReadingContext.target_gender`, and all downstream consumers remain the same.
