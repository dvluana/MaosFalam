# Project Research Summary

**Project:** MaosFalam — Backend Infrastructure
**Domain:** Freemium AI-powered consumer reading app (quiromancy)
**Researched:** 2026-04-10
**Confidence:** HIGH

## Executive Summary

MaosFalam is a freemium mobile-first web app where users photograph their palm and receive a quiromancy reading generated by GPT-4o vision. The backend is built on Next.js 16 App Router (serverless functions), Neon PostgreSQL via Prisma 7, Clerk for auth, and a static TypeScript reading engine (`selectBlocks`). The frontend is complete and already runs against mocks — the backend milestone is replacing those mocks with real API routes, database persistence, and auth. No new stack decisions are needed: all libraries are installed, schemas exist, and the architecture is fully documented.

The recommended build order is strict: database schema and Prisma setup first (all server libs depend on generated types), then infrastructure libs (logger, rate limiter, auth helpers), then middleware, then the AI pipeline and reading engine, then public API routes, then auth-gated routes, and finally client adapter transition. This order minimizes blocked work and surfaces the highest-value flow (photo capture -> reading) as early as possible. Payment (AbacatePay) and email (Resend) are intentionally deferred to a separate milestone.

The key risks are concentrated in three areas: Prisma connection management (missing `DIRECT_URL` breaks migrations; missing singleton pattern exhausts Neon connections), Clerk placement (must live in `proxy.ts` in Next.js 16, not `middleware.ts`), and GPT-4o output handling (`selectBlocks` crashes on unrecognized variations unless hardened with Zod validation and fallback blocks). Credit debit race conditions are a financial risk that requires a single atomic SQL UPDATE, not a read-then-write pattern. All of these are preventable with the documented fixes applied in the right phases.

## Key Findings

### Recommended Stack

The stack is already installed and correct. No new packages are needed. The important versioning details are: Prisma 7.7 (not v6 — import path is `@/generated/prisma/client`, not `@prisma/client`), Clerk 7.0.12 (`auth()` is async, file is `proxy.ts`), Zod 4.3.6 (breaking changes from v3: `z.email()` not `z.string().email()`), and Next.js 16.2.3 (requires Node.js 20.9+, `middleware.ts` deprecated).

**Core technologies:**

- Next.js 16 App Router API routes — serverless functions per route, zero separate server process
- Prisma 7 + `@prisma/adapter-neon` — type-safe DB access, Neon WebSocket driver bundled, no separate `@neondatabase/serverless` needed
- Neon PostgreSQL — serverless Postgres, free tier covers MVP, `DATABASE_URL` (pooled) for runtime, `DIRECT_URL` (direct) for migrations
- Clerk v7 — Google OAuth + email/password, 50K users free, `clerkMiddleware` in `src/proxy.ts`, `await auth()` in routes
- GPT-4o via raw `fetch` — structured JSON output with `response_format: {type: "json_schema"}`, `detail: "high"`, text content before image in array
- Zod v4 — input validation first gate on every API route, v4 syntax required
- Pino — structured logging, never log name/email/CPF, only IDs and action codes

### Expected Features

**Must have (table stakes) — this milestone:**

- Lead capture (POST /api/lead/register) — email recovery and retargeting depend on it
- Photo-to-reading pipeline (POST /api/reading/capture) — core product value
- Reading persistence + GET /api/reading/[id] — share links and session restore require it
- Tier enforcement server-side — free/premium gate enforced in DB, never client-controlled
- Confidence-based rejection — low-confidence GPT-4o responses routed to error screen, not garbage readings
- Clerk auth middleware — gates /conta/\* and auth-required API routes
- Reading history + profile + credits (GET endpoints) — logged-in users expect history and balance
- Account deletion (DELETE) — LGPD compliance
- Rate limiting — 5/h per IP on capture (GPT-4o costs money), 10/h on lead register
- Zod validation everywhere — every route, every input
- Security headers — 6 headers, applied globally
- Photo discarded after processing — never stored, LGPD + trust signal
- Pino logging without PII

**Should have (competitive differentiators — already built or low effort):**

- Gender-concordant reading text — already in `selectBlocks`, makes readings feel personal
- Deterministic text variation per reading — seeded by `reading.id`, same reading = same text
- Reading for another person (POST /api/reading/new with target) — doubles revenue opportunity
- Pre-hint element from MediaPipe landmarks to GPT-4o — improves accuracy, reduces token cost
- Permanent share links (UUID-based, never expire) — persistent acquisition channel

**Defer (v2+):**

- AbacatePay payment integration — deferred to next milestone (blocked on domain + docs confirmation)
- Resend email — deferred to next milestone (blocked on domain verification)
- Upstash rate limiting migration — in-memory Map is MVP-adequate for single instance
- Admin dashboard — build after PMF
- Hand comparison / left vs right / compatibility — requires new DB relationships

### Architecture Approach

The architecture is layered: client adapters (`src/lib/`) call API routes (`src/app/api/`), which call server libs (`src/server/lib/`), which call Neon via Prisma or OpenAI via raw fetch. The boundary between `src/server/lib/` and `src/lib/` is hard — server code never reaches the browser bundle. The reading engine (`selectBlocks`) is a pure TypeScript function with zero I/O and <1ms execution, fed by static block content in `src/data/blocks/`. Report JSON is stored in full on creation; tier upgrade only flips a flag — no recomputation.

**Major components:**

1. `src/middleware.ts` / `src/proxy.ts` — Clerk auth gate, runs before all non-static requests
2. `src/app/api/` — HTTP surface: validate (Zod) → orchestrate → respond
3. `src/server/lib/openai.ts` — GPT-4o wrapper, returns typed `HandAttributes`
4. `src/server/lib/select-blocks.ts` — pure reading engine, `HandAttributes + name + gender → ReportJSON`
5. `src/server/lib/prisma.ts` — singleton Prisma client with Neon adapter
6. `src/lib/reading-client.ts` / `src/lib/user-client.ts` — client adapters replacing mocks
7. Neon PostgreSQL — 5 tables: leads, user_profiles, readings, credit_packs, payments

### Critical Pitfalls

1. **Prisma not singleton** — `new PrismaClient()` outside `globalThis` guard exhausts Neon connections on hot-reload and under concurrent Vercel invocations. Use the existing singleton pattern in `src/server/lib/prisma.ts`.

2. **Missing `DIRECT_URL`** — `prisma migrate deploy` fails against Neon's pooled connection (PgBouncer doesn't support DDL). Add `DIRECT_URL` (unpooled) to env and `directUrl` to `prisma/schema.prisma`. Use pooled `DATABASE_URL` for runtime only.

3. **Clerk in `middleware.ts` instead of `proxy.ts`** — In Next.js 16, `middleware.ts` is deprecated. Clerk must be in `src/proxy.ts`. If placed in `middleware.ts`, auth silently fails and all protected routes become public.

4. **Middleware as sole auth guard** — Every protected API route must call `await auth()` directly inside the handler. Middleware is UX (redirect to login), not security. Never accept `userId` or `tier` from request body.

5. **GPT-4o response crashes `selectBlocks`** — `selectBlocks` throws on unknown variations (documented in CONCERNS.md). Use `response_format: {type: "json_schema"}` with strict schema, parse GPT-4o response through Zod before calling `selectBlocks`, and add fallback blocks for every axis.

6. **Credit debit race condition** — Two concurrent `/api/reading/new` requests can both see `remaining = 1` and double-spend. Use a single atomic `UPDATE ... WHERE remaining > 0 RETURNING id` — never read-then-write for financial operations.

7. **Full-resolution photo payload** — Phone camera photos sent as base64 hit Vercel's 4.5MB body limit and cost 5,200 GPT-4o tokens. Resize to max 1024px, JPEG 0.85 quality client-side before sending. Target payload under 300KB.

## Implications for Roadmap

Based on combined research, the build order from ARCHITECTURE.md is the correct phase structure. The following phases respect strict technical dependencies:

### Phase 1: Database and Infrastructure Foundation

**Rationale:** All server libs depend on Prisma-generated types. Logger is used everywhere. Without these, nothing else can be built. The two DIRECT_URL and singleton pitfalls must be resolved here before any business logic touches the DB.

**Delivers:** Working Neon connection, Prisma migrations deployed, all 5 tables created, infrastructure libs (logger, rate limiter, auth helpers) ready for import.

**Addresses:** Lead persistence, reading persistence, user_profiles, credit_packs, payments tables.

**Avoids:** Pitfall 1 (Prisma singleton), Pitfall 2 (DIRECT_URL / migration failure), Pitfall 8 (Neon cold start — document as known behavior here).

### Phase 2: Auth Middleware

**Rationale:** Protected routes cannot be built until middleware is in place. Middleware depends only on Clerk package and knowledge of which routes to protect. Establishing the `auth()` vs `currentUser()` rule here prevents costly refactors later.

**Delivers:** Clerk `clerkMiddleware` in `src/proxy.ts`, `createRouteMatcher` covering all protected routes, `auth.ts` helpers (getClerkUserId / getClerkUser).

**Addresses:** Auth requirement for /conta/_ and all /api/user/_ and /api/reading/new.

**Avoids:** Pitfall 3 (proxy.ts vs middleware.ts), Pitfall 4 (middleware as sole auth guard), Pitfall 9 (auth() vs currentUser() cost).

### Phase 3: AI Pipeline and Reading Engine

**Rationale:** This is the highest-value flow and the most complex. It depends on Phase 1 (Prisma types for reading persistence) and Phase 2 (auth helpers for logger context). Hardening `selectBlocks` and the GPT-4o wrapper here, before wiring to client, prevents silent failures in production.

**Delivers:** `openai.ts` with structured JSON output, `selectBlocks` hardened with Zod validation and fallback blocks, photo resize enforcement, confidence thresholds routing to correct error states.

**Addresses:** Reading capture core value, confidence-based rejection, reading persistence.

**Avoids:** Pitfall 5 (GPT-4o crashes selectBlocks), Pitfall 6 (photo size / 413 / token cost), Pitfall 10 (selectBlocks no fallback).

### Phase 4: Public API Routes

**Rationale:** With the AI pipeline and DB ready, public routes are wiring calls together. These have no auth dependency. Lead capture and reading capture are the highest-traffic routes and should be validated independently before adding auth-gated routes.

**Delivers:** POST /api/lead/register, POST /api/reading/capture, GET /api/reading/[id]. Rate limiting active. Zod validation on all inputs. Security headers applied globally.

**Addresses:** All table-stakes features for the reading funnel.

**Avoids:** Cost attacks (rate limit on capture), malformed inputs (Zod first gate), PII in logs.

### Phase 5: Auth-Gated API Routes

**Rationale:** These depend on Phase 2 (auth) and Phase 1 (DB). POST /api/reading/new requires credit debit atomicity — implement the single-UPDATE pattern here, not read-then-write.

**Delivers:** POST /api/reading/new (credit debit FIFO), GET /api/user/credits, GET /api/user/readings, GET+PUT /api/user/profile, DELETE /api/user/account.

**Addresses:** Reading history, profile CRUD, credit balance, account deletion (LGPD).

**Avoids:** Pitfall 7 (credit debit race condition — atomic UPDATE required).

### Phase 6: Client Adapter Transition

**Rationale:** Frontend works on mocks now. The adapter transition is the last step because you need stable API contracts first. `useMock()` is the existing toggle mechanism. Replace mock calls in `reading-client.ts` and `user-client.ts` once API routes are verified.

**Delivers:** Real data flowing through the frontend. `useMock()` removed from production path. End-to-end integration tested against real Neon.

**Addresses:** Frontend mocks replaced by real API.

**Avoids:** Anti-Pattern 1 (report re-fetched from DB — use response state, not second GET).

### Phase 7: Payment Integration (Next Milestone — Deferred)

**Rationale:** Explicitly out of scope for this milestone. AbacatePay integration requires domain setup and webhook docs confirmation. Builds on Phase 5 (credit_packs schema already exists).

**Delivers:** POST /api/credits/purchase, POST /api/webhook/abacatepay, tier flip on payment confirmation.

**Addresses:** Monetization, tier upgrade from free to premium.

**Note:** Webhook idempotency (unique constraint on `abacatepay_billing_id`) must be the first thing implemented here.

### Phase 8: Email (Next Milestone — Deferred)

**Rationale:** Blocked on domain verification with Resend. Builds on Phase 4 (fire-and-forget from capture route). Fire-and-forget pattern already documented.

**Delivers:** Transactional emails: lead created, payment confirmed, leitura pronta.

### Phase Ordering Rationale

- Prisma-generated types are a hard dependency for every server lib — schema must come first.
- Logger is used in every route and lib — infrastructure before business logic.
- Public routes before auth routes — capture (highest value) is testable without auth.
- Client adapters last — frontend works on mocks and only switches after API is stable.
- Payment and email deferred — they have external dependencies (domain, AbacatePay docs) that are not in this milestone's control.

### Research Flags

Phases likely needing deeper research during planning:

- **Phase 7 (Payment):** AbacatePay v2 webhook signature validation is under-documented. Research the exact HMAC algorithm and header names before implementation. Also verify idempotency key support.
- **Phase 3 (GPT-4o structured outputs):** OpenAI Structured Outputs with `json_schema` mode requires a JSON Schema representation of `HandAttributes`. This schema must be derived from `src/types/` and kept in sync — the tooling for this is non-trivial.

Phases with standard patterns (skip research-phase):

- **Phase 1 (Database):** Neon + Prisma 7 setup is fully documented in official docs with exact patterns. STACK.md has the complete verified config.
- **Phase 2 (Auth):** Clerk v7 with Next.js 16 is documented. PITFALLS.md has the exact proxy.ts pattern.
- **Phase 4-5 (API routes):** Standard Next.js App Router patterns. Well-documented. Zod + Prisma patterns are established.
- **Phase 6 (Client adapters):** Frontend structure already exists. No research needed — swap mock calls for fetch calls.

### Gaps to Address

- **GPT-4o JSON Schema for HandAttributes:** The Structured Outputs API requires a JSON Schema (not Zod schema) representation of `HandAttributes`. This must be built from `src/types/hand-attributes.ts` and maintained in sync. No tooling is currently set up for this — handle in Phase 3 planning.
- **`selectBlocks` fallback blocks not yet defined:** CONCERNS.md flags `selectBlocks` as fragile for unknown variations. Fallback block content needs to be written for each axis (`_fallback` key). Handle in Phase 3.
- **`proxy.ts` vs `middleware.ts` current state:** STACK.md notes both files exist in the codebase. The correct state for Next.js 16 is Clerk in `proxy.ts`. Verify and consolidate in Phase 2.
- **`@neondatabase/serverless` redundant package:** Listed as a separate dependency in `package.json` but bundled inside `@prisma/adapter-neon`. Remove when convenient — not blocking but adds package weight.
- **In-memory rate limiter production behavior:** Vercel runs isolated function instances. In-memory Map rate limits are per-instance and ineffective in production with concurrent requests. This is a known technical debt item. Plan Upstash migration before public launch (v1.x).
