# Quick Task 260411-p9a: Linking leituras anônimas a contas

## Goal

Vincular leituras anônimas à conta quando usuária se cadastra. Gate de repeat reading para visitantes. Feedback de email já cadastrado.

## Context

- Schema já suporta: Lead.clerkUserId, Reading.clerkUserId existem mas nunca são populados para anônimos
- /api/reading/capture salva readings com leadId mas sem clerkUserId
- /api/user/readings busca só por clerkUserId → leituras anônimas ficam órfãs
- Nenhuma migration necessária

## Plan 1: Claim readings API + hook integration

### Task 1: POST /api/user/claim-readings

**files:** src/app/api/user/claim-readings/route.ts (NEW)
**action:**

- Create new API route (auth required via getClerkUserId + getClerkUser)
- Get user email from Clerk
- Find leads WHERE email = clerkEmail AND clerkUserId IS NULL
- Update matching leads: set clerkUserId, converted = true
- Find readings WHERE leadId IN (matched lead IDs) AND clerkUserId IS NULL
- Update those readings: set clerkUserId
- Also find readings by sessionId if available (belt and suspenders)
- Return { claimed_count: N }
- Idempotent: calling multiple times is safe (WHERE clerkUserId IS NULL prevents double-claiming)
  **verify:** Build passes, route responds 200 with claimed_count
  **done:** Orphaned readings get linked to account by email match

### Task 2: Lead register — check for existing Clerk account

**files:** src/app/api/lead/register/route.ts
**action:**

- After Zod validation, before creating lead:
- Import clerkClient from @clerk/nextjs/server
- Call clerkClient.users.getUserList({ emailAddress: [data.email] })
- If users.length > 0: return { existing_account: true, lead_id: null } with status 200 (not error)
- If no existing account: proceed with normal lead creation
  **verify:** Build passes, existing email returns existing_account flag
  **done:** Frontend can detect existing accounts

### Task 3: useAuth hook — call claim-readings on first login

**files:** src/hooks/useAuth.ts
**action:**

- Add logic: when user transitions from null → authenticated (first hydration with user)
- Check localStorage flag `maosfalam_readings_claimed`
- If not set: POST /api/user/claim-readings (fire-and-forget, failure non-blocking)
- On success: set localStorage flag so it doesn't run again
  **verify:** Build passes, hook calls claim on first login
  **done:** Readings automatically linked after registration

### Task 4: /ler/nome — gate repeat anonymous readings + email feedback

**files:** src/app/ler/nome/page.tsx
**action:**

- In handleVisitorSubmit, after registerLead call resolves:
  - If response has existing_account: true → show inline message "Esse email já tem conta. Faz login." + highlight login button, don't navigate
- For repeat reading gate:
  - After successful reading (when scan completes), sessionStorage already has `maosfalam_reading_id`
  - In NomePage, check if sessionStorage has `maosfalam_reading_id` AND user is visitor
  - If so: show message "Crie uma conta pra fazer mais leituras" + buttons to /registro and /login
  - The registerLead is fire-and-forget so we need to await it for the email check
  - Change from fire-and-forget to awaited (with try/catch, still non-blocking for navigation if it fails)
    **verify:** Build passes, visitor with existing reading sees gate, existing email shows feedback
    **done:** Anonymous users gated after first reading, existing emails get feedback

### Task 5: /api/user/readings — also fetch by leadId

**files:** src/app/api/user/readings/route.ts
**action:**

- After fetching readings by clerkUserId, also fetch user's email from Clerk
- Find leads by that email
- Fetch readings WHERE leadId IN (those lead IDs) AND clerkUserId IS NULL (unclaimed stragglers)
- Merge both sets, deduplicate by reading.id, sort by createdAt desc
- This is a safety net: claim-readings should catch most, but this catches edge cases
  **verify:** Build passes, readings include both claimed and unclaimed
  **done:** Dashboard shows all readings regardless of claim status
